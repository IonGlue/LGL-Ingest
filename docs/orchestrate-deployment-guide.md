# LGL Ingest — Orchestrate Deployment Guide

How orchestrate deploys and manages per-tenant ingest stacks.

## Container Architecture

Each tenant gets 5 containers:

| Container | Image | Network | Purpose |
|-----------|-------|---------|---------|
| `postgres` | `postgres:16-alpine` | Internal | Database |
| `redis` | `redis:7-alpine` | Internal | Pub/sub, device channels |
| `ingest-supervisor` | `ghcr.io/ionglue/ingest-supervisor:latest` | **Host** | Spawns GStreamer workers, manages ports |
| `ingest-api` | `ghcr.io/ionglue/ingest:latest` | Internal | REST API + WebSocket + serves frontend |
| `cloudflared` | `cloudflare/cloudflared:latest` | Internal | Tunnel for HTTP(S) access |

The supervisor uses host networking because it binds to dynamic ports for video streams (SRT, RTMP). All other containers use internal networking.

## Images to Build & Push

### 1. Ingest API (includes frontend)

Built from `server/Dockerfile`. This is a multi-stage build that:
- Builds the React frontend with Vite
- Compiles the TypeScript server
- Bundles both into a single Node.js image serving static files and the API

```
ghcr.io/ionglue/ingest:latest
```

CI already builds this on push to `main` (`.github/workflows/build.yml`).

### 2. Ingest Supervisor

Built from `Dockerfile.supervisor`. Multi-stage Rust build that produces 4 binaries:
- `ingest-supervisor` — process manager and REST API
- `ingest-source` — source worker (SRT, RTMP, test pattern)
- `ingest-dest` — destination worker (RTMP push, SRT push, HLS, recorder)
- `ingest-sync` — multi-camera frame sync worker

```
ghcr.io/ionglue/ingest-supervisor:latest
```

CI needs to be added for this image — currently only the API image is built in the workflow.

## Per-Tenant Configuration

### Supervisor Config (generated by orchestrate)

Write to a file and mount at `/etc/ingest-supervisor/config.toml`:

```toml
[server]
listen = "0.0.0.0:{port_api}"

[ports]
internal_srt_start = {internal_srt_start}
internal_srt_end   = {internal_srt_end}

[storage]
recordings = "/recordings"
hls        = "/var/hls"
tmp        = "/tmp"
```

Where `{port_api}`, `{internal_srt_start}`, `{internal_srt_end}` are allocated per tenant by orchestrate's port manager.

### Ingest API Environment Variables

```env
# Database
DATABASE_URL=postgres://lgl:lgl@postgres:5432/lgl_ingest

# Redis
REDIS_URL=redis://redis:6379

# Auth
JWT_SECRET={generated_per_tenant}

# Supervisor connection
SUPERVISOR_API_URL=http://host.docker.internal:{port_api}

# Config file path
CONFIG_PATH=/app/config/ingest.toml

# Orchestrate integration (Logto auth mode)
ORCH_API_URL={orchestrate_backend_url}
INTERNAL_SECRET={internal_secret}
TENANT_SLUG={tenant_slug}
LOGTO_JWKS_URL=https://auth.lgl-os.com/oidc/jwks
LOGTO_ISSUER=https://auth.lgl-os.com/oidc
LOGTO_API_RESOURCE=https://api.orchestrate.lgl-os.com
```

On Linux hosts, add `extra_hosts: ["host.docker.internal:host-gateway"]` so the API container can reach the host-networked supervisor.

### Ingest API Config (optional override)

If you want to override the TS server defaults, mount a TOML config at `CONFIG_PATH`:

```toml
[server]
host = "0.0.0.0"
port = 3000
ws_path = "/ws/device"

[database]
url = "postgres://lgl:lgl@postgres:5432/lgl_ingest"
max_connections = 20

[redis]
url = "redis://redis:6379"

[auth]
jwt_secret = "{generated}"
device_token_ttl = 3600
user_token_ttl = 86400

[supervisor]
api_url = "http://host.docker.internal:{port_api}"
```

In practice, env vars override most of these, so this file is optional.

### PostgreSQL

Standard setup, no special config:

```env
POSTGRES_DB=lgl_ingest
POSTGRES_USER=lgl
POSTGRES_PASSWORD=lgl
```

Migrations run automatically on API server startup.

## Port Allocation

Orchestrate needs to allocate these port ranges per tenant:

| Resource | Count | Example Range | Protocol | Exposed |
|----------|-------|---------------|----------|---------|
| Supervisor API | 1 port | 30010 | TCP/HTTP | Internal only |
| Internal SRT routing | ~200 ports | 10000-10199 | UDP | Internal only |
| RTMP ingest | ~10 ports | 1935-1944 | TCP | External (via tunnel or firewall) |
| SRT ingest | ~100 ports | 5000-5099 | UDP | External (via tunnel or firewall) |

The internal SRT range and supervisor API port are configured in the supervisor's `config.toml`. 

The RTMP and SRT ingest ports are **not** managed by the supervisor — they're specified per-source when the user creates an ingest source via the API. Orchestrate should define the valid range and pass it to the frontend or API so users pick ports within their allocated range.

## Docker Compose Template

Example template for orchestrate to generate per tenant:

```yaml
version: '3.9'

services:
  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: lgl_ingest
      POSTGRES_USER: lgl
      POSTGRES_PASSWORD: lgl
    volumes:
      - postgres_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    restart: unless-stopped

  supervisor:
    image: ghcr.io/ionglue/ingest-supervisor:latest
    restart: unless-stopped
    network_mode: host
    volumes:
      - /tmp:/tmp
      - recordings:/recordings
      - hls:/var/hls
      - ./config/supervisor.toml:/etc/ingest-supervisor/config.toml:ro
    environment:
      RUST_LOG: info

  ingest-api:
    image: ghcr.io/ionglue/ingest:latest
    restart: unless-stopped
    depends_on:
      - postgres
      - redis
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://lgl:lgl@postgres:5432/lgl_ingest
      REDIS_URL: redis://redis:6379
      JWT_SECRET: {{JWT_SECRET}}
      SUPERVISOR_API_URL: http://host.docker.internal:{{PORT_API}}
      ORCH_API_URL: {{ORCH_API_URL}}
      INTERNAL_SECRET: {{INTERNAL_SECRET}}
      TENANT_SLUG: {{TENANT_SLUG}}
      LOGTO_JWKS_URL: https://auth.lgl-os.com/oidc/jwks
      LOGTO_ISSUER: https://auth.lgl-os.com/oidc
      LOGTO_API_RESOURCE: https://api.orchestrate.lgl-os.com
    extra_hosts:
      - "host.docker.internal:host-gateway"

  cloudflared:
    image: cloudflare/cloudflared:latest
    restart: unless-stopped
    command: tunnel run
    environment:
      TUNNEL_TOKEN: {{CF_TUNNEL_TOKEN}}

volumes:
  postgres_data:
  recordings:
  hls:
```

## Startup Order

1. **postgres** + **redis** — start first, no dependencies
2. **supervisor** — starts, binds API port, ready to accept source/dest registrations
3. **ingest-api** — connects to postgres (runs migrations), redis, and supervisor
4. **cloudflared** — establishes tunnel for external access

## Health Checks

| Service | Endpoint | Expected |
|---------|----------|----------|
| Supervisor | `GET http://host:{port_api}/health` | `{"status":"ok","sources":N,...}` |
| Ingest API | `GET http://host:3000/api/system/stats` | JSON with uptime, counts |
| PostgreSQL | `pg_isready -U lgl` | Exit code 0 |
| Redis | `redis-cli ping` | `PONG` |

## Video Flow (End-to-End)

```
User creates source in UI (e.g., RTMP Listen on port 1935)
  → API POST /api/sources {source_type: "rtmp_listen", config: {port: 1935}}
  → API auto-generates stream key (UUID)
  → API registers source with supervisor (POST supervisor:port_api/sources)
  → Supervisor allocates internal SRT port from pool (e.g., 10042)
  → User copies ingest URL: rtmp://tenant.example.com:1935/live/{stream_key}

User creates destination (e.g., YouTube RTMP)
  → API POST /api/destinations {dest_type: "rtmp", config: {url: "rtmp://a.rtmp.youtube.com/live2/KEY"}}

User routes source → destination in Rack UI
  → API POST /api/routing {source_id, dest_id}

User starts source
  → Supervisor spawns ingest-source worker (RTMP server on port 1935, SRT out on 10042)

User starts destination (or it auto-starts when source is active)
  → Supervisor spawns ingest-dest worker (SRT in from 10042, RTMP out to YouTube)

Encoder (OBS) pushes to rtmp://tenant.example.com:1935/live/{stream_key}
  → ingest-source validates key, accepts stream
  → Video relayed: RTMP in → demux → SRT internal → mux → RTMP out to YouTube
```

## CI/CD: Adding Supervisor Build

The current workflow only builds the API image. Add a second job for the supervisor:

```yaml
  build-supervisor:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ghcr.io/ionglue/ingest-supervisor
          tags: |
            type=raw,value=latest
            type=sha
      - uses: docker/build-push-action@v5
        with:
          context: .
          file: Dockerfile.supervisor
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

Note: The supervisor build is slow (~5-10 min) because it compiles the full Rust workspace. Consider caching `target/` or using a Cargo chef layer for faster incremental builds.
